## 3. Decisions Made

| Decision | Rationale |
|----------|-----------|
| Add `option_placement` as command metadata with default `"any"` | This matches the challenge requirement while preserving permissive behavior for normal commands. |
| Mark `run` as `option_placement="strict"` | It behaves like a wrapper/forwarding command and is the clearest strict-placement exception in the current CLI. |
| Enforce strict placement by validating original argv after parsing resolves the command | This keeps the default parser behavior intact for normal commands while allowing strict commands to reject misplaced framework or command options. |
| Include `option_placement` in command-level `--schema` output | Agents may introspect a single command rather than the full app schema, so the metadata needs to be visible there too. |
| Commit only the four code/test files related to the fix | There were unrelated untracked files in the workspace, so the git scope was kept intentionally narrow. |
| Use branch name `fix/strict-option-placement` | Creating a `codex/...` branch hit a ref-creation problem, so a plain fix branch kept the task moving without touching git internals. |

## 4. Decisions Refused / Alternatives Rejected

| Rejected Approach | Reason |
|-------------------|--------|
| Treat `option_placement` as schema-only metadata | The review correctly identified that this would mislead command authors because strict commands would not actually reject bad argument order. |
| Leave command-level `--schema` unchanged | That would hide strict-placement metadata from agents that inspect only one command. |
| Commit or push unrelated untracked files such as `.claude/`, `.githooks/`, or `docs/root_global_options_evaluation.md` | They were not part of the fix and would have widened the change set unnecessarily. |
| Force the original `codex/...` branch prefix by manually manipulating `.git` internals | The branch creation issue was low-level and not worth risking repository state when a normal fix branch solved the task cleanly. |
